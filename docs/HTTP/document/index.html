HTTP

reference from: https://notfalse.net/37/http-resource-representation
採用Uniform Interface

「HTTP 透過 Client 發送 請求 (Request)，Server 送出 回應 (Response) 以進行互動。」

HTTP request包含#

  1. Start-line:請求方法-URL-協議版本 (極重要)
  2. 請求標頭Header
  3. 請求正文Payload

HTTP response包含#

  1. status-line: HTTP版本-狀態碼(status-code)-原因短語(reason-phrash)及CRLF
  2. 響應標頭Header
  3. 響應正文payload

請求方法#

  1. GET: 請求展示指定資源，只應用於取得資料。 (給我)
  • 敏感資料不應使用GET
  1. POST: 提交指定資源的實體，通常會改變伺服器狀態。
  2. PATH: 套用指定資源的部分修改。
  3. DELETE: 刪除指定資源。

http request(example)#

from fastapi import FastAPI
app = FastAPI()
@app.get("/")async def root():    return {"message": "hello aaron"}
  1. 代碼的中間部分（以 開頭的三行app.get）顯示了一個路由定義。該方法指定了一個回調函數，只要有一個帶有相對於站點根目錄的路徑 ( ) 的 HTTP 請求app.get()，就會調用該回調函數。回調函數將請求和響應對像作為參數，並調用響應以返回字符串“Hello World！”GET'/'send()

HTTP資源、表示、URL#

  1. http資源代表是某個特定的目標當下的資料。
  2. http response格式，包含json, xml
  3. http 資源狀態：又稱為表示方式、文件、檔案、http訊息實體(entity)、實例(instance)
  4. http meatadata(元資料)：為描述資料資訊的資料
  • 就像是一本書的資訊是內文，而meatadata包含『出版社、作者、ISBN等等』

Representation#

  • representation header fields
  • http representation 通常包含：內容種類(Content-Type)、內容編碼(Content-Encoding)、內容語言(Content-Language)、內容位置(Content-Location)、host(不可省)

Content-Type#

  • 內容種類(Content-Type：是指表示媒體的類型，定義資料格式，方便接收者以相應的方式處理。 example: application/json; charset=utf-8
  • 主要類型 (type) 通常分為 文字 (text)、影像 (image)、 音訊 (audio)、影片 (video) 及 應用 (application) 五大類，常見如: text/html text/html; charset=utf-8 text/plain; text/css image/jpeg mage/png audio/mpeg audio/ogg video/quicktime video/mp4 application/octet-stream //未知的應用程序文件 application/x-www-form-urlencoded; //常用POST提交帳密 application/x-www-form-urlencoded; charset=utf-8 //常用POST提交帳密 application/json application/xml multipart/form-data; boundary=----XXXXXX //常用於POST表單上傳檔案 message/http
  • 不論是request或response只要有playload，元資料就要帶Content-Type。
  • pplication/octet-stream是未知的應用程序文件，所以不會帶Content-Type。

Content-Encoding#

  • Content-Encoding是編碼方式
  • 若representation有使用壓縮，則必須帶Content-Encoding
  • 常見的編碼方式有 gzip、deflate、br (Brotli) 及 compress
  • 則 (訊息) 發送方 必須生成 Content-Encoding 表頭欄位，並按照 使用順序 列出編碼方式 (coding)。
  • 可多壓縮方式

Content-Language#

  • representation表示文件用於哪個語系的人，但不代表是用那個語系
  • 可多語系TAG

Content-Location#

  • Content-Location 表頭欄位 是指與回應 表示 相對應的最具體資源，可能是備用 URI，也可能是更具體的位址，主要用途是做為 內容協商 的資源 URI。
  • 值得注意的是，另一表頭欄位: Location (位置)，是回應訊息中，引用與 回應相關 的特定資源 之 URI，主要用於 — — 重新導向。有些導致資源狀態改變的 請求/回應 (e.g., 以 POST 方法新增資源後的 201 Created 回應)，Content-Location 表頭欄位 與 Location (位置) 意義相近，皆指向新建立的資源，否則，兩者大多時候意義不同。

訊息格式#

  • Message是HTTP/1.1的最小傳輸單元
  • 組成：start-line、http header(表頭）、CRLF及message-body(訊息本體optional)

CRLF#

  • 是接續在 表頭 (Header) 之後的 必要 部分。然而，許多粗心的程設設計師，在沒有 訊息主體 (message body) 部分時，時常忽略了 CRLF，是實作上需注意的地方。

Http status code#

401vs403#

401: 代表身份驗證錯誤。 403: 是沒有訪問此資源的權限錯誤。

playload#

不要求格式編碼等等

統一資源識別符(URI)#

  • Uniform Resource Identifier, URI
  • 辨識 資源 的方式」，為 REST 架構風格 中的四個介面約束之一
  • URI可細分成 URL及URN

URI Schema#

http(s) URI = "http(s):" "//" authority path-abempty [ "?" query ][ "#" fragment ] Ex: https://echo.paw.cloud:443/hello/world?age=24&gender=female#Uehara

念法#

  • 冒號 (:) 唸作 colon
  • 斜線 (/) 唸作 slash
  • 點 (.) 唸作 dot
  • 井號 (#) 唸作 hash

reference from: https://notfalse.net/36/http-uri

info

authority: 具有『權限』送出回應者，為主機名稱+tcp port path-abempty: 可以為空，或是以 “/” 開始的路徑。 [ "?" query ]查詢 (query)， 以「?」開始，並接著 key=value 的鍵值對 (中間別忘了加 =)，若還有其他 query ，則通常以「&」做為區隔，是提供給 Server 參數的方式之一。 譬如: ?age=24&gender=female代表的可能是，我想在這個網站 (如果有支援)尋找:『年紀 24 且 性別是女性 』。

需要注意的是:

如果 query 字串中，包含 保留字元 (Reserved)， 例如 「@」、「:」、「/」、「?」、「%」、「=」、「 」(空白) … ，必須使用 "%" HEXDIG HEXDIG HEXDIG 代表的是 hexadecimal digits，以 UTF-8 編碼的 十六進制 位數， 例如: 「:」表示為 「%3A」，這是 URI 規範中的 百分比編碼 (Percent-Encoding)，也就是俗稱的 URI (URL) 編碼。

若保留字元是 「 」(空白)，其百分比編碼表示為: 「%20」，可替代為「+」 。 其他特殊字元，如中文 (大部分為 3個位元組)，譬如: 「勝」表示為「%E5%8B%9D」， 大部分瀏覽器網址列允許中文，是因為它幫你做好了❗️ (複製下來在別處貼上，則會顯示 百分比編碼) 為了一致性，十六進制的英文部分，以『大寫』為主 正常形式不需進行編碼

danger

開發時，需 記得處理 query 字串 (尤其是 客戶端應用)， 以免送出保留字元或錯誤的編碼。

如 JavaScript:

var result = encodeURIComponent(query);
